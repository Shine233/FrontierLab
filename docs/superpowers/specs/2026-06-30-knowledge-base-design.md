# FrontierLab 知识库建设设计

> 状态：设计已确认，待写实现计划。
> 日期：2026-06-30

## 目标

把现有的前沿论文阅读/对比/复现 markdown 库，升级成一个可视化、可索引、可学习、可标准化产出、有跨会话记忆的知识库。在不破坏现有约定（`papers/<direction>/<year>-<slug>.md` + YAML frontmatter + 结论先行 + 证据/推断分离）的前提下增量扩展。

## 总体架构：数据同源

唯一事实源是每篇论文的 YAML frontmatter。其余一切（静态索引、Obsidian 动态视图、时间线、谱系图）都从 frontmatter 派生，不重复维护。

```
论文 frontmatter (唯一事实源)
        │
        ├── scripts/build_indices.py ──► indices/*.md   (静态, GitHub 可读)
        ├── dashboards/*.md (Dataview)  ──► Obsidian 动态视图
        └── 正文「方法谱系」markdown 链接 ──► Obsidian Graph view 连边
```

GitHub 和 Obsidian 两端读同一份数据，满足「两者兼顾」。

## 五个模块

### 模块 1：Frontmatter 扩展

在 `papers/_template.md` 现有字段基础上新增 5 个字段（全部可选，留空不影响现有论文，脚本对缺失字段降级处理）：

| 字段 | 类型 | 用途 |
|---|---|---|
| `landmark` | bool | 标记划时代/奠基方法。时间线和 landmarks 索引据此加 ★ |
| `supersedes` | list[slug] | 取代/改进的前作。谱系图实线边 |
| `builds_on` | list[slug] | 依赖但不取代的前作（如用 DINOv2 backbone）。谱系图虚线边 |
| `org` | list[str] | 机构/团队。回答「哪个 lab/公司在领跑某方向」 |
| `key_idea` | str | 一句话核心。索引表 Notes 列、Obsidian 悬浮预览复用 |

不新增的轴及原因：`compute_tier`（主观，复现 README 已记成本）、`modality`（`tasks` 已含）、`paradigm`（`method_family` 已含）、`rating`（`confidence` 已含）。`year` / `direction` 已存在，时间线和方向分布直接复用。

### 模块 2：索引生成脚本 + 静态索引

`scripts/build_indices.py`，仅依赖 `pyyaml`。扫描 `papers/**/*.md` frontmatter，幂等全量重写 `indices/` 下文件。

生成文件：

| 文件 | 内容 | 新增? |
|---|---|---|
| `indices/papers.md` | 论文总表（结构化部分自动生成） | 改为半自动 |
| `indices/directions.md` | 按方向分组（结构化部分自动生成） | 改为半自动 |
| `indices/methods.md` | 按 `method_family` 分组 | 改为半自动 |
| `indices/timeline.md` | 按 `year` 倒序，每年列论文 + ★landmark + 该年方向分布小计 | 新增 |
| `indices/landmarks.md` | 仅 landmark 方法 + supersedes/builds_on 链 | 新增 |
| `indices/comparisons.md` | 保持手写，脚本仅在末尾追加对比报告文件清单 | 半自动 |

**保留-标记块机制**：现有 `indices/papers.md` / `directions.md` 含大量人工叙述（关联、复现状态等）。脚本只生成结构化表格，人工叙述用 `<!-- MANUAL:start -->...<!-- MANUAL:end -->` 包裹，脚本不碰块内内容。自动表格与人工叙述共存。

所有自动生成文件顶部加：`<!-- AUTO-GENERATED by scripts/build_indices.py, 勿手改自动区 -->`。

### 模块 3：Obsidian 动态视图

仓库根新建 `dashboards/`（Obsidian 专用GitHub 上是含代码块的普通 md，无害）。Dataview 查询页，直接读 frontmatter，与脚本同源：

- `dashboards/timeline.md` — `TABLE year, direction ... SORT year DESC`
- `dashboards/landmarks.md` — `WHERE landmark = true`
- `dashboards/by-direction.md` — 按方向 group
- `dashboards/reproduction-status.md` — 按复现状态筛

**谱系连边**：论文正文加「方法谱系」小节，用标准 markdown 链接（非 `[[wikilink]]`）：

```markdown
## 方法谱系
- 取代/改进：[RoMa](../papers/image-matching/2024-roma.md)
- 基于：[DINOv2](../papers/vision-foundation-models/2023-dinov2.md)
```

标准 markdown 链接在 GitHub 可点，且 Obsidian Graph view 也将其识别为图的边——一份语法两端通用。谱系信息写两处（frontmatter 给脚本，正文给图），冗余换两生态原生可用；skill 生成时两处一起写。

### 模块 4：Skills（标准化流程）

放仓库根 `skills/`，每个 skill 一个目录含 `SKILL.md`，作为人/AI 均可读的 SOP 文档库（GitHub 可见），不强依赖 Claude Code 自动发现。

| Skill | 输入 → 产物 | 优先级 |
|---|---|---|
| `skills/paper-read/SKILL.md` | 链接/PDF/arxiv id → `papers/<direction>/<year>-<slug>.md`（含扩展 frontmatter + 正文谱系小节）→ 收尾跑索引脚本 | 1（最高频） |
| `skills/paper-compare/SKILL.md` | 同方向 ≥2 篇 → 生成/更新 `comparisons/<direction>/<topic>.md`，复用稳定维度表 → 收尾跑索引脚本 | 2 |
| `skills/repro-plan/SKILL.md` | 论文 repo → `reproductions/<direction>/<method>/README.md`（环境/commit/最小复现路径/是否值得复现）→ 收尾跑索引脚本 | 3 |

**索引同步不做独立 skill**：降级为 `scripts/build_indices.py`，由上述三个 skill 收尾调用。

每个 skill 运行先读 `KB-MEMORY.md`（模块 5），收尾更新它。

### 模块 5：学习路线图

`learning/<direction>.md`，每方向一份，手写为主。`learning/_template.md` 固定结构：

```markdown
## 0. 这个方向在解决什么（一段话）
## 1. 前置知识（链到外部资源）
## 2. 奠基论文（链到仓库 papers/，标 ★landmark）
## 3. 近期 SOTA 与分支（链到 comparisons/）
## 4. 动手：最小复现路径（链到 reproductions/）
## 5. 常见误区 / 我的判断
```

工作流：AI 按模板为**有论文的 7 个方向**各写一版通用初稿（3d-reconstruction、visual-localization、image-matching、vision-foundation-models、self-supervised-learning、world-models、efficient-training-inference），链到仓库已有内容；用户再用个人经验/项目实战打磨成「宝典」。空方向（multimodal-learning、reasoning-agents 等）等有论文再补。

每份 frontmatter 加 `maturity: draft`（AI 初稿）/ `refined`（用户打磨后），一眼区分成熟度。本模块不做成 skill；初稿在实现阶段批量生成。

### 模块 6：项目记忆文件

`KB-MEMORY.md` 放仓库根。三个 skill 每次先读、收尾更新。结构：

```markdown
## 约定与偏好（命名、风格、对比维度的稳定口径）
## 各方向缺口（还缺哪几篇关键论文）
## 未完成意图（planned 但没做的复现、想写的对比）
## 近期决策记录（为什么这么分类/判断）
```

分工：索引 = 结构化事实（机读）；`KB-MEMORY.md` = 叙事性决策和待办（给下次的人和 AI 看）。

## 实现顺序

1. 模块 1 frontmatter 扩展（改 `_template.md` + README 元数据说明）
2. 模块 2 `build_indices.py` + 保留-标记块改造现有索引 + 两个新索引
3. 模块 3 `dashboards/` 四个 Dataview 页
4. 模块 5 `learning/_template.md` + 7 个方向初稿
5. 模块 6 `KB-MEMORY.md` 初版
6. 模块 4 三个 `SKILL.md`（最后做，因为它们引用前面所有产物和脚本）

## 非目标（YAGNI）

- 不做 index-sync 独立 skill
- 不给空方向写学习路线
- 不引入除 pyyaml 外的依赖
- 不做 frontmatter 增量 diff（脚本全量幂等重写）
- 不提交大文件（数据集/权重/缓存只记路径，沿用现有约定）

## 风险

- 现有 `indices/papers.md` / `directions.md` 改造时若标记块划分不当，可能误删人工叙述 → 改造前先备份，逐文件核对。
- 谱系信息两处冗余可能不同步 → skill 生成时强制两处一起写；脚本可加校验：frontmatter 的 supersedes slug 在正文谱系小节是否有对应链接。
